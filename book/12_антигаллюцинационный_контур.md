## 12.1. Почему агенту нужен ревьюер

### Принцип разделения ролей

Генератор оптимизирован на **fluency** (гладкость текста). Ревьюер оптимизирован на **accuracy** (точность содержания). Совмещать обе функции в одном вызове — значит требовать от модели одновременно быть «креативной» и «критичной», что создаёт конфликт в attention-паттернах.

### Эмпирические данные

Разделение генерации и проверки снижает hallucination rate на **40–70%** по данным экспериментов с CoVe (Dhuliawala et al., 2023) и двухэтапных pipeline'ов (2024–2025).

### Архитектура: Generator + Verifier

```
┌──────────────┐     ┌──────────────┐     ┌──────────────┐
│  Generator   │ ──→ │   Output     │ ──→ │  Verifier    │
│  (high temp) │     │  (candidate) │     │  (low temp)  │
└──────────────┘     └──────────────┘     └──────┬───────┘
                                                  │
                                           ┌──────┴───────┐
                                           │              │
                                           ▼              ▼
                                      [PASS]         [FAIL]
                                        │               │
                                        ▼               ▼
                                   [Return]      [Retry / Fix]
```

**Generator**: температура 0.3–0.7, emphasis on coverage и creativity.
**Verifier**: температура 0.0–0.1, emphasis on accuracy и consistency.

Это могут быть:
- Два вызова одной модели с разными системными промптами.
- Две разные модели (generator = GPT-4o, verifier = Claude Opus).
- Одна модель + внешние проверки (тесты, linter, schema validation).

## 12.2. Chain-of-Verification (CoVe)

### Метод (Dhuliawala et al., 2023)

CoVe — четырёхэтапный протокол, специально разработанный для снижения галлюцинаций:

**Этап 1: Draft (Черновик)**
```
Промпт: "Перечисли 5 крупнейших озёр Африки по площади."
Черновик: "1. Виктория, 2. Танганьика, 3. Малави, 4. Чад, 5. Туркана"
```

**Этап 2: Plan Verification Questions (Планирование проверки)**
```
Промпт: "Для каждого факта сформулируй проверочный вопрос."
Вопросы:
- "Виктория — одно из крупнейших озёр Африки?"
- "Танганьика — одно из крупнейших озёр Африки?"
- "Чад — одно из крупнейших озёр Африки по площади?"
- ...
```

**Этап 3: Answer Questions Independently (Независимые ответы)**

Критически важно: проверочные вопросы отвечаются **в отдельном контексте**, без видимости черновика. Это предотвращает confirmation bias.

```
"Чад — одно из крупнейших озёр Африки по площади?"
→ "Озеро Чад значительно сократилось и не входит в топ-5 по текущей площади.
   Альберт (Albert) — крупнее."
```

**Этап 4: Generate Verified Response (Финальный ответ)**
```
Исправленный ответ:
"1. Виктория, 2. Танганьика, 3. Малави, 4. Туркана, 5. Альберт"
```

### Почему CoVe работает

1. **Декомпозиция проверки**: вместо «проверь всё» — отдельные вопросы для каждого факта.
2. **Независимость**: ответы на проверочные вопросы не contaminated черновиком.
3. **Итеративность**: можно запустить несколько раундов CoVe для повышения точности.

### Реализация на Python

```python
async def chain_of_verification(question: str, model) -> dict:
    # Этап 1: Черновик
    draft = await model.generate(
        f"Answer this question: {question}",
        temperature=0.3
    )
    
    # Этап 2: Проверочные вопросы
    verification_questions = await model.generate(
        f"""Draft answer: {draft}
        Generate specific factual questions to verify each claim in the draft.
        Return as JSON list: ["question1", "question2", ...]""",
        temperature=0.1,
        response_format={"type": "json_object"}
    )
    
    # Этап 3: Независимые ответы
    verifications = []
    for vq in json.loads(verification_questions):
        # Новый контекст — без видимости черновика!
        answer = await model.generate(
            f"Answer factually: {vq}",
            temperature=0.0
        )
        verifications.append({"question": vq, "answer": answer})
    
    # Этап 4: Финальный ответ
    final = await model.generate(
        f"""Original question: {question}
        Draft: {draft}
        Verification results: {json.dumps(verifications)}
        Generate corrected final answer based on verifications.""",
        temperature=0.1
    )
    
    return {
        "draft": draft,
        "verifications": verifications,
        "final_answer": final
    }
```

## 12.3. Log-Driven Development (LDD)

### Принцип: данные вместо интуиции

Традиционный подход к промптингу: написал → попробовал → переписал → попробовал... Это метод тыка, маскирующийся под итерацию.

**LDD** — структурированный подход:

1. **Логируй всё**: промпт, параметры, output, validation, метрики.
2. **Анализируй**: где ошибки? На каком шаге? Какой тип?
3. **Формируй гипотезу**: «ошибка возникает, когда данные >5K токенов».
4. **Тестируй**: измени один параметр, проверь на тех же данных.
5. **Итерируй**: на основе данных, а не чутья.

### Что логировать

```python
@dataclass
class LLMCallLog:
    # Идентификация
    call_id: str
    timestamp: datetime
    pipeline_name: str
    step_name: str
    
    # Вход
    system_prompt: str
    user_prompt: str
    model: str
    temperature: float
    max_tokens: int
    tools: list[dict] | None
    
    # Выход
    response: str
    tool_calls: list[dict] | None
    finish_reason: str  # "stop", "length", "tool_calls"
    
    # Метрики
    input_tokens: int
    output_tokens: int
    latency_ms: int
    cost_usd: float
    
    # Валидация
    schema_valid: bool
    factual_checks: list[dict] | None
    human_rating: int | None  # 1-5, заполняется позже
    
    # Контекст
    error: str | None
    retry_count: int
```

### Метрики для отслеживания

| Метрика | Формула | Цель |
|---------|---------|------|
| **Schema compliance rate** | Valid outputs / Total outputs | >99% |
| **Hallucination rate** | Verified false claims / Total claims | <5% |
| **Retry rate** | Retries / Total calls | <10% |
| **Latency P95** | 95-й перцентиль response time | <3s |
| **Cost per task** | Σ(tokens × price) / tasks | Зависит от задачи |
| **Success rate** | Tasks completed / Tasks attempted | >90% |

### Инструменты для LDD

| Инструмент | Назначение | Тип |
|-----------|------------|-----|
| **LangSmith** (LangChain) | Tracing, debugging, evaluation | SaaS |
| **Weights & Biases (Prompts)** | Tracking, versioning | SaaS |
| **Braintrust** | Evaluation, logging | SaaS |
| **Phoenix (Arize)** | Observability, traces | Open-source |
| **OpenTelemetry + custom** | DIY tracing | Open-source |
| **SQLite + JSON logs** | Минимальный локальный | DIY |

## 12.4. Anti-Loop Protocol: детекция зацикливания

### Паттерны зацикливания

**1. Вербальные петли**: модель повторяет одну и ту же фразу или абзац.

```
"Для решения этой задачи необходимо рассмотреть все аспекты. 
Рассмотрев все аспекты, мы можем заключить, что для решения 
необходимо рассмотреть все аспекты..."
```

**2. Tool-call петли**: агент вызывает одни и те же инструменты с одинаковыми аргументами.

```
Action: search_files("config.json")
Observation: Not found
Action: search_files("config.json")  ← повтор!
Observation: Not found
Action: search_files("config.json")  ← повтор!
```

**3. State oscillation**: агент переключается между двумя состояниями без прогресса.

```
State: "needs_fix" → fix_code() → test_fails → "needs_fix" → fix_code() → test_fails → ...
```

**4. Output inflation**: модель генерирует всё более длинные ответы, разбавляя содержание.

### Детекция

```python
class LoopDetector:
    def __init__(self, max_repeats: int = 3, similarity_threshold: float = 0.9):
        self.history: list[str] = []
        self.tool_history: list[tuple[str, str]] = []  # (name, args)
        self.max_repeats = max_repeats
        self.threshold = similarity_threshold
    
    def check_verbal_loop(self, output: str) -> bool:
        """Детектирует повторяющиеся фрагменты в выводе."""
        sentences = output.split(". ")
        seen = {}
        for s in sentences:
            normalized = s.strip().lower()
            seen[normalized] = seen.get(normalized, 0) + 1
            if seen[normalized] >= self.max_repeats:
                return True
        return False
    
    def check_tool_loop(self, tool_name: str, tool_args: str) -> bool:
        """Детектирует повторяющиеся tool-вызовы."""
        call = (tool_name, tool_args)
        self.tool_history.append(call)
        
        # Последние N вызовов одинаковы?
        recent = self.tool_history[-self.max_repeats:]
        if len(recent) == self.max_repeats and len(set(recent)) == 1:
            return True
        return False
    
    def check_progress(self, state: dict) -> bool:
        """Детектирует отсутствие прогресса."""
        state_str = json.dumps(state, sort_keys=True)
        self.history.append(state_str)
        
        # Состояние не менялось N шагов?
        recent = self.history[-self.max_repeats:]
        if len(recent) == self.max_repeats and len(set(recent)) == 1:
            return True
        return False
```

### Реакция на зацикливание

| Тип петли | Действие |
|-----------|----------|
| Вербальная | Прервать генерацию, вернуть частичный результат |
| Tool-call | Изменить стратегию (другой tool, другие аргументы) |
| State oscillation | Откат к предыдущему стабильному состоянию + переформулировка |
| Output inflation | Установить жёсткий max_tokens |
| Все типы (после retry) | **Human-in-the-loop**: запросить помощь оператора |

## 12.5. Guardrails: защитные барьеры

### Входные guardrails

```python
def validate_input(user_input: str) -> tuple[bool, str]:
    """Проверка входных данных перед отправкой в модель."""
    # 1. Длина
    if count_tokens(user_input) > MAX_INPUT_TOKENS:
        return False, "Input too long"
    
    # 2. Prompt injection detection
    injection_patterns = [
        r"ignore previous instructions",
        r"you are now",
        r"system:\s*",
        r"<\|.*\|>",  # ChatML injection
    ]
    for pattern in injection_patterns:
        if re.search(pattern, user_input, re.IGNORECASE):
            return False, "Potential prompt injection detected"
    
    # 3. PII detection (simplified)
    if re.search(r"\b\d{3}-\d{2}-\d{4}\b", user_input):  # SSN
        return False, "PII detected: SSN-like pattern"
    
    return True, "ok"
```

### Выходные guardrails

```python
def validate_output(output: str, schema: dict | None = None) -> tuple[bool, list[str]]:
    """Проверка выхода модели перед возвратом пользователю."""
    issues = []
    
    # 1. Schema validation
    if schema:
        try:
            jsonschema.validate(json.loads(output), schema)
        except (json.JSONDecodeError, jsonschema.ValidationError) as e:
            issues.append(f"Schema validation failed: {e}")
    
    # 2. Toxicity check
    if toxicity_score(output) > TOXICITY_THRESHOLD:
        issues.append("Output flagged for toxicity")
    
    # 3. Confidence check
    hedging_phrases = ["я не уверен", "возможно", "примерно", "наверное"]
    hedge_count = sum(1 for p in hedging_phrases if p in output.lower())
    if hedge_count > 3:
        issues.append(f"Low confidence: {hedge_count} hedging phrases")
    
    # 4. Hallucination markers
    if re.search(r"as of my (last|knowledge) (cutoff|update)", output, re.I):
        issues.append("Model referencing training cutoff — may be outdated")
    
    return len(issues) == 0, issues
```

## 12.6. Практический вывод

### Минимальный антигаллюцинационный контур

```
Input → [Input Guardrails] → [Generator] → [Output Guardrails] →
   → [Verifier (CoVe / tests / schema)] → Output / Retry
```

### Чек-лист

| # | Компонент | Внедрено? |
|---|-----------|-----------|
| 1 | **Ревьюер отделён от генератора** | Generator ≠ Verifier |
| 2 | **CoVe для фактуальных задач** | 4-step protocol |
| 3 | **Логи всех вызовов** | Промпт, output, метрики, validation |
| 4 | **Детекторы петель** | Verbal, tool-call, state oscillation |
| 5 | **Input guardrails** | Длина, injection, PII |
| 6 | **Output guardrails** | Schema, toxicity, confidence |
| 7 | **Метрики** | Schema compliance, hallucination rate, retry rate |
| 8 | **Human-in-the-loop** | Есть fallback к человеку? |
