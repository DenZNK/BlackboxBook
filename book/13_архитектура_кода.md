## 13.1. Маленькие модули лучше монолита

### Почему модель ошибается в монолитном коде

LLM обрабатывает код как последовательность токенов с ограниченным context window. Монолитный файл в 2000 строк создаёт несколько проблем:

**1. Attention dilution**: в файле на 2000 строк (~6000–10000 токенов) attention распределяется на множество функций, классов, import'ов. Когда модель модифицирует функцию на строке 1500, она «видит» функцию на строке 50, но attention к ней минимален (Lost in the Middle).

**2. Implicit dependencies**: в монолите зависимости скрыты. Функция `process_order()` на строке 800 может зависеть от глобальной переменной на строке 12, helper'а на строке 300 и класса на строке 600. Модель должна отследить все эти связи — Transformer может это делать, но каждая дополнительная связь увеличивает вероятность ошибки.

**3. Context overflow**: если файл не помещается в context window — модель работает с фрагментом. Изменение одного фрагмента без видимости другого → несогласованности.

### Оптимальный размер модуля

| Размер файла | Токены (Python) | Качество генерации | Рекомендация |
|-------------|-----------------|---------------------|-------------|
| <100 строк | ~300–500 | ✅ Отлично | **Идеал** |
| 100–300 строк | 500–1500 | ✅ Хорошо | Приемлемо |
| 300–500 строк | 1500–2500 | ⚠️ Средне | Рефакторить |
| 500+ строк | 2500+ | ❌ Ненадёжно | Разбить обязательно |

**Правило**: один модуль ≤ 100 строк, одна функция ≤ 30 строк, одна ответственность.

### Как это выглядит на практике

**До (монолит):**
```python
# order_service.py — 800 строк
class OrderService:
    def create_order(self, ...): ...      # 100 строк
    def validate_order(self, ...): ...    # 60 строк  
    def calculate_price(self, ...): ...   # 80 строк
    def apply_discount(self, ...): ...    # 40 строк
    def process_payment(self, ...): ...   # 120 строк
    def send_notification(self, ...): ... # 50 строк
    def update_inventory(self, ...): ...  # 70 строк
    # + helpers, constants, error handlers...
```

**После (модули):**
```
order/
├── __init__.py
├── models.py          # 40 строк: Order, OrderItem, OrderStatus
├── validation.py      # 60 строк: validate_order()
├── pricing.py         # 80 строк: calculate_price(), apply_discount()
├── payment.py         # 70 строк: process_payment()
├── notifications.py   # 50 строк: send_notification()
├── inventory.py       # 50 строк: update_inventory()
└── service.py         # 60 строк: create_order() — оркестрация
```

Каждый файл помещается в один промпт с запасом. Модель может модифицировать `pricing.py`, не загружая весь rest of the service.

## 13.2. Контракт функции как фиксатор смысла

### Зачем нужен контракт

Контракт функции — это **семантический якорь** для модели (Глава 1). Типы, docstrings, pre/post-conditions — всё это фиксирует, что функция делает, не заставляя модель угадывать.

### Минимальный контракт

```python
def calculate_order_total(
    items: list[OrderItem],
    discount_percent: float = 0.0,
    tax_rate: float = 0.20,
) -> OrderTotal:
    """Calculate total order price with discount and tax.
    
    Args:
        items: Non-empty list of order items.
        discount_percent: Discount as fraction [0.0, 1.0]. Applied before tax.
        tax_rate: Tax rate as fraction [0.0, 1.0]. Applied after discount.
    
    Returns:
        OrderTotal with subtotal, discount, tax, and total.
    
    Raises:
        ValueError: If items is empty or discount/tax out of [0, 1] range.
    
    Example:
        >>> items = [OrderItem("widget", quantity=3, unit_price=10.0)]
        >>> result = calculate_order_total(items, discount_percent=0.1, tax_rate=0.2)
        >>> result.total
        32.4  # (30 - 3) * 1.2
    """
```

### Что даёт контракт модели

| Элемент | Что фиксирует для модели | Без контракта |
|---------|--------------------------|---------------|
| **Имя функции** | Семантика (calculate = вычислить) | Угадывает из контекста |
| **Типы аргументов** | Формат входных данных | Может принять str вместо float |
| **Типы возврата** | Формат выходных данных | Может вернуть dict вместо dataclass |
| **Docstring** | Полная спецификация поведения | Угадывает по имени |
| **Raises** | Граничные условия | Может проигнорировать ошибки |
| **Example** | Конкретный I/O для калибровки | Нет якоря для формата |

### Pre/Post-conditions и Design by Contract

```python
from typing import assert_type

def transfer_funds(
    from_account: Account,
    to_account: Account, 
    amount: Decimal,
) -> TransferResult:
    """Transfer funds between accounts.
    
    Pre-conditions:
        - amount > 0
        - from_account.balance >= amount
        - from_account.id != to_account.id
        - both accounts are active
    
    Post-conditions:
        - from_account.balance decreased by amount
        - to_account.balance increased by amount
        - Transaction logged with timestamp
    
    Invariants:
        - Total balance across all accounts unchanged
    """
    # Pre-conditions (executable)
    assert amount > Decimal("0"), "Amount must be positive"
    assert from_account.balance >= amount, "Insufficient funds"
    assert from_account.id != to_account.id, "Cannot transfer to self"
    
    # ... implementation ...
    
    # Post-conditions (executable)
    assert from_account.balance == old_from_balance - amount
    assert to_account.balance == old_to_balance + amount
```

Модель видит pre/post-conditions и генерирует код, который **по конструкции** их соблюдает. Без них — каждое ограничение нужно угадывать.

## 13.3. Zero-Context Survival: код должен работать без промпта

### Тест на автономность

Код, сгенерированный ИИ, должен быть **полностью самодокументированным** — работать и быть понятным без оригинального промпта. Это критерий качества:

**Вопросы для проверки:**
1. Может ли новый разработчик понять функцию, прочитав только код?
2. Все ли зависимости явны (imports, типы, конфигурация)?
3. Понятны ли границы ответственности (что функция делает и чего НЕ делает)?
4. Есть ли обработка ошибок для реальных сценариев?

### Антипаттерн: контекстно-зависимый код

```python
# ❌ Требует контекст промпта для понимания
def process(data):
    result = []
    for item in data:
        if item[2] > threshold:  # Какой threshold? Что такое item[2]?
            result.append(transform(item))  # Какая transform?
    return result
```

```python
# ✅ Zero-Context Survival
from dataclasses import dataclass
from typing import Sequence

@dataclass(frozen=True)
class SensorReading:
    sensor_id: str
    timestamp: float
    temperature_celsius: float

def filter_high_temperature_readings(
    readings: Sequence[SensorReading],
    threshold_celsius: float = 40.0,
) -> list[SensorReading]:
    """Return readings where temperature exceeds threshold."""
    return [r for r in readings if r.temperature_celsius > threshold_celsius]
```

## 13.4. "Small Simple Blocks": линейный код лучше переинжиниренного DRY

### Проблема чрезмерной абстракции

DRY (Don't Repeat Yourself) — полезный принцип, но его чрезмерное применение создаёт проблемы для AI-генерации:

**Переабстрагированный код:**
```python
# ❌ Сложно для модели: стратегии, фабрики, абстрактные базовые классы
class PricingStrategy(ABC):
    @abstractmethod
    def calculate(self, items: list[Item]) -> Decimal: ...

class StandardPricing(PricingStrategy):
    def calculate(self, items): ...

class DiscountPricing(PricingStrategy):
    def __init__(self, discount_factory: DiscountFactory): ...
    def calculate(self, items):
        discount = self.discount_factory.create(items)
        return discount.apply(StandardPricing().calculate(items))

class DiscountFactory:
    def create(self, items: list[Item]) -> Discount:
        if self._is_bulk(items):
            return BulkDiscount(self.config)
        return NoDiscount()
```

**Линейный код:**
```python
# ✅ Просто для модели: линейный, явный, self-documenting
def calculate_order_price(
    items: list[OrderItem],
    discount_percent: float = 0.0,
) -> Decimal:
    """Calculate total price with optional discount."""
    subtotal = sum(item.quantity * item.unit_price for item in items)
    discount = subtotal * Decimal(str(discount_percent))
    return subtotal - discount
```

### Почему линейный код лучше для AI

1. **Меньше indirection**: модель не должна отслеживать цепочку `Strategy → Factory → Discount → apply()`.
2. **Все шаги видимы**: в линейном коде каждый шаг — одна строка. В абстрактном — каждый шаг спрятан за interface.
3. **Модификация проще**: чтобы изменить расчёт, достаточно изменить одну функцию, а не всю иерархию.
4. **Тестирование проще**: один вход → один выход, без моков и dependency injection.

### Когда абстракция оправдана

| Критерий | Линейный код | Абстракция |
|----------|-------------|------------|
| Вариантов < 3 | ✅ if/elif | ❌ Strategy pattern |
| Код используется 1–2 раза | ✅ Дублирование | ❌ Helper function |
| Логика < 30 строк | ✅ Inline | ❌ Отдельный класс |
| Требования стабильны | ✅ Прямой код | ⚠️ Зависит |
| Требования часто меняются | ⚠️ Зависит | ✅ Абстракция с контрактом |
| Вариантов > 5 | ❌ Спагетти | ✅ Strategy/Registry |

**Правило WET (Write Everything Twice)**: дублируйте код до тех пор, пока не увидите три одинаковых фрагмента. Тогда — и только тогда — абстрагируйте. Преждевременная абстракция опаснее дублирования.

## 13.5. Как писать код, который AI легко модифицирует

### Принципы AI-friendly кода

**1. Явные зависимости через аргументы (не globals):**
```python
# ❌ Скрытая зависимость
def get_user():
    return db.query(User).filter_by(id=current_user_id).first()

# ✅ Явная зависимость
def get_user(db: Session, user_id: int) -> User | None:
    return db.query(User).filter_by(id=user_id).first()
```

**2. Один файл = одна концепция:**
```
# ❌ models.py — 500 строк с User, Order, Product, Payment, Notification

# ✅ 
models/
├── user.py       # User, UserRole
├── order.py      # Order, OrderItem, OrderStatus  
├── product.py    # Product, Category
├── payment.py    # Payment, PaymentMethod
└── notification.py # Notification, NotificationType
```

**3. Конфигурация через переменные, не через код:**
```python
# ❌ Magic numbers
if retry_count > 3 and elapsed > 30:
    raise TimeoutError()

# ✅ Named configuration
MAX_RETRIES = 3
TIMEOUT_SECONDS = 30

if retry_count > MAX_RETRIES and elapsed > TIMEOUT_SECONDS:
    raise TimeoutError(f"Exceeded {MAX_RETRIES} retries in {TIMEOUT_SECONDS}s")
```

**4. Тесты рядом с кодом:**
```
pricing/
├── __init__.py
├── calculator.py
├── test_calculator.py    # Рядом с кодом, а не в отдельной папке tests/
└── conftest.py
```

Модель видит тест рядом с реализацией → может обновить оба одновременно.

## 13.6. Практический вывод

### Чек-лист AI-friendly кода

| # | Правило | Метрика |
|---|---------|---------|
| 1 | **Модули < 100 строк** | `wc -l` на каждый файл |
| 2 | **Функции < 30 строк** | Cyclomatic complexity < 10 |
| 3 | **Явные контракты** | Типы + docstring + примеры |
| 4 | **Zero-Context Survival** | Новичок поймёт без промпта? |
| 5 | **Линейный > абстрактный** | WET до 3-х повторений |
| 6 | **Явные зависимости** | Через аргументы, не globals |
| 7 | **Тесты рядом** | `test_*.py` в том же каталоге |
| 8 | **Нет magic numbers** | Именованные константы |

### Ментальная модель

> **Пишите код как документацию к себе.** Если через год вам (или модели) нужно будет изменить функцию — все контракты, зависимости и ограничения должны быть видимы в файле, а не храниться в чьей-то голове или промпт-истории.
