## 15.1. Конец эпохи чистого Transformer

### Что произошло

На протяжении 2017–2024 годов Transformer (Vaswani et al., 2017) был единственной доминирующей архитектурой для языковых моделей. Весь прогресс шёл через масштабирование: больше параметров, больше данных, больше compute.

К 2025–2026 годам стало очевидно, что **чистый Transformer упирается в потолок эффективности**. Квадратичная сложность attention ($O(n^2)$ по длине последовательности) делает обработку очень длинных контекстов непрактично дорогой. FlashAttention и sparse attention помогают, но не устраняют фундаментальную проблему.

Представьте аэропорт. Если каждый пассажир должен поздороваться с каждым другим пассажиром (это Transformer attention) — при 100 пассажирах это 10 000 рукопожатий, при 1000 — миллион. Никакая оптимизация рукопожатий не сделает эту систему масштабируемой. Нужна **другая система** — например, информационное табло, где каждый пассажир просто проверяет свой рейс (это SSM — линейная сложность).

### State Space Models: линейная альтернатива

**State Space Models (SSM)**, в частности **Mamba** (Gu & Dao, 2023), предлагают принципиально другой подход к обработке последовательностей:

| Свойство | Transformer (Attention) | Mamba (SSM) |
|----------|------------------------|-------------|
| **Сложность по длине** | $O(n^2)$ | $O(n)$ — линейная |
| **Throughput** | Базовый | 3–10× выше |
| **Latency** | Базовая | 2–3× ниже |
| **Точное извлечение** | ✅ Отлично (attention) | ⚠️ Хуже на point lookups |
| **Длинные зависимости** | ⚠️ Degradation | ✅ Стабильно |
| **KV-кэш** | Растёт с длиной | Фиксированный state |

**Как работает Mamba (интуиция для не-ML инженеров)**:

Представьте конвейерную ленту на фабрике. Изделие движется по ленте, и на каждой станции к нему что-то добавляется или изменяется. Каждая станция «помнит» только текущее состояние изделия и свои инструкции — ей не нужно видеть все предыдущие станции. Это **рекуррентный** подход: фиксированный state обновляется на каждом шаге.

В отличие от attention (где каждый токен «смотрит» на все предыдущие), SSM поддерживает **скрытое состояние** (hidden state) фиксированного размера, которое сжимает всю историю в компактный вектор.

**Ключевая инновация Mamba**: **селективные механизмы** — параметры, управляющие обновлением состояния, зависят от входных данных. Это позволяет модели решать, какую информацию сохранить, а какую забыть — динамически, в зависимости от контекста.

### Гибридные архитектуры: лучшее из двух миров

Главный тренд 2025–2026 — **объединение attention и SSM в одной модели**:

| Модель | Организация | Архитектура | Пропорция | Результат |
|--------|-------------|-------------|-----------|-----------|
| **Jamba** | AI21 Labs | Attention + Mamba | 1:7 (attention:Mamba) | Первая production-гибридная модель |
| **Nemotron-3 Super** | NVIDIA | Mamba + Latent MoE | 75% Mamba | 3× throughput при сопоставимом качестве |
| **Phi-4-mini-flash-reasoning** | Microsoft | Mamba + Attention | 75% Mamba-слоёв | Reasoning на edge-устройствах |
| **Qwen3-Next** | Alibaba | Gated DeltaNet + Attention | Гибрид | Новое поколение attention |
| **Llama 4 Maverick** | Meta | Attention + early-fusion MoE | 128 экспертов | 1M+ контекст |

**Принцип гибрида**: 
- **SSM/Mamba слои** обрабатывают длинные последовательности эффективно (O(n)), создавая общий контекст
- **Attention слои** выполняют точное извлечение и сопоставление (когда нужно найти конкретный факт в контексте)
- Комбинация даёт скорость SSM + точность attention

Это как организация библиотеки: SSM — каталог, который позволяет быстро найти нужный раздел, а attention — чтение конкретных страниц.

## 15.2. MoE: больше не экзотика, а стандарт

### Почему MoE победил

К 2026 году **более 60% frontier-моделей используют Mixture of Experts** (MoE). Причина проста: MoE позволяет иметь огромный объём знаний (параметров) при разумных затратах на inference.

| Модель | Всего параметров | Активных на токен | Экономия |
|--------|-----------------|-------------------|----------|
| **DeepSeek-V3.2** | 671B | 37B | 94.5% параметров «спят» |
| **Mixtral 8×7B** | 46.7B | 12.9B | 72% |
| **Llama 4 Maverick** | ~400B+ | ~120B | ~70% |
| **Qwen3-235B** | 235B | ~22B | ~91% |

**Аналогия**: MoE — это корпорация, а не один суперсотрудник. Вместо одного человека, который знает всё (dense model), у вас 256 специалистов, каждый эксперт в своей области. На каждый запрос дежурная система (router) вызывает 2–8 нужных специалистов, остальные не тратят ресурсы.

### DeepSeek-V3.2: как обучить frontier-модель за «копейки»

DeepSeek-V3.2 — наглядная демонстрация эффективности MoE:

- **671B параметров**, но активируются только **37B** на каждый токен
- Качество сопоставимо с GPT-5 при **90% меньших затратах на обучение**
- Используют **auxiliary-loss-free load balancing** — решение проблемы неравномерной загрузки экспертов
- **Multi-Token Prediction (MTP)** — модель предсказывает несколько следующих токенов одновременно → ускорение inference

**Что это значит для практики**: если вы разворачиваете модели на собственной инфраструктуре, MoE-модели дают frontier-качество при значительно меньших требованиях к GPU (потому что за раз активируется малая доля параметров).

## 15.3. Diffusion LLMs: революция или любопытство?

### Что это такое

Diffusion LLMs — концептуально новый подход к генерации текста, заимствованный из мира генерации изображений (Stable Diffusion, DALL-E).

**Классический подход (авторегрессионный)**:
```
Генерация: token₁ → token₂ → token₃ → ... → tokenₙ
(последовательно, слева направо)
```

**Diffusion подход**:
```
Начало: [шум] [шум] [шум] [шум] [шум] [шум]
Шаг 1:  [Пр..] [шум] [ив..] [шум] [ми..] [шум]
Шаг 2:  [Прив] [ет,] [мир] [!]   [Как] [де..]
Шаг 3:  [Привет,] [мир!] [Как] [дела?]
(параллельно, все позиции одновременно)
```

Аналогия: авторегрессионная модель — писатель, который пишет текст слева направо, буква за буквой, никогда не возвращаясь. Diffusion модель — скульптор, который видит весь блок мрамора и постепенно проявляет форму из хаоса, работая над всеми частями одновременно.

### Ключевые модели

| Модель | Организация | Скорость | Особенности |
|--------|-------------|----------|-------------|
| **LLaDA** | — | Сопоставимая | Первая Diffusion LLM уровня GPT-2 |
| **Gemini Diffusion** | Google | **1479 токенов/сек** | Рекордная скорость генерации |

### Почему это важно

1. **Reversal curse решён**: авторегрессионные модели плохо делают «обратные» задачи (знают «A → B», но не «B → A»). Diffusion модели видят весь текст целиком и не имеют этой проблемы.

2. **Параллельная генерация**: все токены генерируются одновременно, а не один за другим. Потенциал для **на порядок более быстрого inference** (1479 токенов/сек у Gemini Diffusion vs типичные 50–100 у авторегрессионных моделей).

3. **Планирование**: модель может «видеть» конец предложения до того, как начнёт его генерировать. Это делает текст более когерентным.

### Ограничения (честная оценка, 2026)

- **Качество**: пока уступает лучшим авторегрессионным моделям на большинстве бенчмарков
- **Тренировка**: требует новых подходов к обучению, несовместимо с RLHF/DPO в текущем виде
- **Зрелость**: технология на ранней стадии, не готова для production

**Прогноз**: Diffusion LLMs — не замена Transformer в 2026, но важная R&D-ветка. Через 2–3 года возможна конвергенция (гибриды авторегрессивных и diffusion подходов, как сейчас гибриды attention + SSM).

## 15.4. Reasoning-модели: Test-Time Compute как новое измерение

### Что такое reasoning-модели

Reasoning-модели — это LLM с **встроенным test-time compute**: они могут «думать дольше» над сложными задачами, используя больше вычислительных ресурсов на этапе inference.

| Модель | Организация | Год | Характеристика |
|--------|-------------|-----|----------------|
| **o1 / o3** | OpenAI | 2024–2025 | Первые mainstream reasoning-модели |
| **Claude Opus 4.6** | Anthropic | 2026 | Extended thinking, 200K контекст |
| **GPT-5 Pro** | OpenAI | 2026 | Reasoning + multimodality |
| **Grok 4** | xAI | 2026 | Training-free reasoning |
| **Gemini 3 Pro** | Google | 2026 | Reasoning + 2M контекст |

### T² Scaling Laws: революция 2026

Классические **Scaling Laws** (Kaplan et al., 2020; Hoffmann/Chinchilla, 2022) говорили: «больше параметров + больше данных = лучше».

**T² Scaling** (Test-Time Scaling, Snell et al., 2024) добавляет **третье измерение**: compute на inference.

$$\text{Quality} = f(\text{Training Compute}, \text{Model Size}, \text{Inference Compute})$$

**Ключевой инсайт**: оптимально **overtrain** меньшую модель (дать ей больше данных, чем предлагает Chinchilla) и компенсировать размер за счёт **inference compute** (extended thinking, самопроверка, множественная генерация).

**Аналогия**: представьте экзамен. Студент A учился 4 года и решает задачу за 5 минут. Студент B учился 2 года, но ему дали 30 минут — он проверяет решение, пробует разные подходы, находит ошибки. Результат одинаковый, но студент B стоил дешевле в «обучении».

### Что это значит на практике

| Подход | Стоимость обучения | Стоимость inference | Качество |
|--------|-------------------|---------------------|----------|
| Большая dense модель | $$$$ | $$ | Высокое |
| Маленькая модель + T² | $ | $$$ | Сопоставимое |
| Маленькая модель, быстрый ответ | $ | $ | Среднее |

**Практическое правило**: для задач, где latency не критична (анализ, планирование, код-ревью), меньшая модель с extended thinking часто выгоднее, чем большая модель с мгновенным ответом.

### Формы test-time compute

1. **Extended thinking** (o3, Claude Opus 4.6): модель генерирует внутреннюю цепочку рассуждений перед ответом. Пользователь видит только финальный ответ (или «thinking» блок).

2. **Self-verification**: модель генерирует ответ, затем проверяет его и исправляет ошибки.

3. **Multiple sampling + voting**: генерация N вариантов, выбор лучшего (Self-Consistency, Best-of-N).

4. **Tree search**: систематическое исследование пространства решений (LATS, Monte Carlo Tree Search).

5. **Tool-augmented reasoning**: модель использует инструменты (калькулятор, поиск, код) как часть рассуждения.

## 15.5. Мультимодальность: текст — это только начало

### Новая норма

К 2026 году мультимодальность стала **дефолтом**, а не опцией:

| Модель | Входные модальности | Выходные модальности |
|--------|---------------------|----------------------|
| **GPT-5** | Текст, изображения, аудио, видео | Текст, изображения, аудио |
| **Claude Opus 4.6** | Текст, изображения, PDF | Текст |
| **Gemini 2.5 Pro** | Текст, изображения, аудио, видео, код | Текст, изображения |
| **Llama 4 Maverick** | Текст, изображения | Текст |

### Что это значит для инженера

1. **Промпты теперь включают изображения**: «Проанализируй этот скриншот и напиши код для такого же UI» — реальный production use case.

2. **Токенизация расширяется**: изображение разбивается на патчи (обычно 16×16 или 14×14 пикселей), каждый патч → вектор → «визуальный токен». Одно изображение ≈ 500–2000 токенов. Это входит в тот же контекстный бюджет.

3. **Мультимодальные галлюцинации**: модель может «видеть» на изображении то, чего нет. Верификация становится ещё важнее.

4. **Генеративные медиа**: Sora 2, Veo 3 (Google), Wan (Alibaba) — генерация видео с синхронизированным аудио. World models (Marble от World Labs, Genie 3 от DeepMind) создают интерактивные 3D-среды из одного промпта.

### Для наших целей

Все принципы из этой книги — протоколы, верификация, декомпозиция, инструменты — применимы к мультимодальным системам. Просто помните:
- Изображения «съедают» контекстный бюджет быстро
- Мультимодальные галлюцинации требуют мультимодальной верификации
- Промпт для vision-задач следует тем же принципам структурирования (Глава 6–7)

## 15.6. Open Source vs Closed Source: сближение

### Текущий баланс (апрель 2026)

```
Closed source:  GPT-5, Claude Opus 4.6, Gemini 3 Pro, Grok 4
Open source:    Llama 4, DeepSeek-V3.2, Qwen3, Mistral Large 2
```

**Ключевые факты:**
- Open-source модели **почти догнали** closed-source по качеству на большинстве бенчмарков
- Open-source **значительно опережает** в кастомизации (fine-tuning, domain adaptation)
- **58% компаний** выбирают AI-инфраструктуру в первую очередь по стоимости (a16z, 2025)
- Предприятия предпочитают fine-tuning своих данных на open-source вместо отправки данных в closed API

### Когда что выбирать

| Критерий | Closed API | Open Source (self-hosted) |
|----------|-----------|--------------------------|
| **Быстрый старт** | ✅ Регистрация + API key | ❌ Нужна инфраструктура |
| **Максимальное качество** | ✅ Frontier модели | ⚠️ Близко, но немного отстают |
| **Стоимость (маленький объём)** | ✅ Pay-per-use | ❌ GPU дорого при малых объёмах |
| **Стоимость (большой объём)** | ❌ Растёт линейно | ✅ Фикс за GPU, масштабируется |
| **Кастомизация** | ❌ Только prompt engineering | ✅ Fine-tuning, LoRA, адаптация |
| **Data sovereignty** | ❌ Данные уходят в API | ✅ Всё на вашем железе |
| **Latency** | ⚠️ Сеть + очередь | ✅ Контроль полный |
| **Compliance (GDPR, etc.)** | ⚠️ Сложнее | ✅ Полный контроль |

### Hybrid deployment: production reality

Самый распространённый подход в 2026 — **гибридный**:

```
┌─────────────────────────────────────────────┐
│              Production система               │
│                                               │
│  Рутинные задачи (80%)  │  Сложные задачи (20%)│
│  Self-hosted:           │  Cloud API:          │
│  • Llama 4 Scout (17B)  │  • Claude Opus 4.6   │
│  • Qwen3-8B             │  • GPT-5 Pro         │
│  • DeepSeek-V3.2        │  • Gemini 3 Pro      │
│  via vLLM/Ollama        │                      │
│                         │                      │
│  ✅ Дёшево              │  ✅ Максимум качества │
│  ✅ Быстро              │  ⚠️ Дороже           │
│  ✅ Data sovereignty    │  ⚠️ Данные уходят    │
└─────────────────────────────────────────────┘
```

**Инструменты для self-hosting:**
- **vLLM** — высокопроизводительный inference server (PagedAttention, continuous batching)
- **llm-d** — Kubernetes-native serving с disaggregated prefill/decode
- **Ollama** — простейший self-hosting для разработки
- **TGI** (HuggingFace) — production inference с streaming

## 15.7. Экономика inference: главный вызов 2026

### Проблема масштаба

**Inference demand уже в 118× превышает training demand** (a16z, 2025). Это самая быстрорастущая статья расходов в AI.

Reasoning-модели усугубляют проблему: extended thinking может использовать 10–100× больше токенов на запрос, чем обычный ответ.

### Стратегии оптимизации

| Стратегия | Экономия | Сложность | Описание |
|-----------|----------|-----------|----------|
| **Prompt caching** | 50–90% | Низкая | Кэшировать длинные system prompts (Anthropic, OpenAI) |
| **Model routing** | 40–70% | Средняя | Простые запросы → дешёвая модель, сложные → дорогая |
| **Quantization** | 2–4× | Средняя | 4-bit/8-bit весов при минимальной потере качества |
| **KV-cache compression** | 2–4× | Средняя | GQA, MQA, KV-cache quantization |
| **Batching** | 3–8× | Средняя | Continuous batching (vLLM), grouped requests |
| **Distillation** | 5–20× | Высокая | Обучить маленькую модель на выходах большой |
| **Speculative decoding** | 1.5–3× | Средняя | Маленькая модель генерирует черновик, большая проверяет |

### Пример: сколько стоит агентная система

Типичный агентный контур для code review (1 PR = ~5000 токенов контекста):

```
Planner:      1 вызов × 3000 output tokens
Executor:     3 шага  × 2000 output tokens  
Verifier:     1 вызов × 1000 output tokens
Total input:  ~20K tokens
Total output: ~10K tokens
```

| Модель | Input cost | Output cost | Total per PR | 1000 PR/мес |
|--------|-----------|-------------|--------------|-------------|
| Claude Opus 4 | $15/1M | $75/1M | ~$0.95 | **$950** |
| GPT-4o | $2.5/1M | $10/1M | ~$0.15 | **$150** |
| Claude Haiku 3.5 | $0.80/1M | $4/1M | ~$0.056 | **$56** |
| Llama 4 (self-hosted) | ~$0.01/1M | ~$0.05/1M | ~$0.001 | **$1** |

**Вывод**: выбор модели определяет стоимость на 2–3 порядка. Используйте дорогие модели только для задач, где это оправдано.

## 15.8. SLM и Edge AI: большое на маленьком

### Efficient модели

Отдельный тренд — **Small Language Models (SLM)**, оптимизированные для работы на edge-устройствах:

| Модель | Параметры | Где работает | Для чего |
|--------|-----------|-------------|----------|
| **Phi-4-mini** | 3.8B | Ноутбук, телефон | Reasoning, код |
| **Llama 4 Scout** | 17B active (MoE) | Десктоп GPU | Универсальные задачи |
| **Qwen3-8B** | 8B | GPU, NPU | Быстрые задачи |
| **Gemma 3** | 2B–12B | Телефон, IoT | On-device AI |

### Почему это важно

1. **Latency**: нет сетевого round-trip → ответ за миллисекунды
2. **Privacy**: данные не покидают устройство
3. **Cost**: нет API-расходов после покупки железа
4. **Offline**: работает без интернета

### Distillation + Quantization

Техники сжатия позволяют запускать модели, которые раньше требовали datacenter, на обычном ноутбуке:

- **Distillation**: маленькая модель обучается имитировать выходы большой. DeepSeek-R1-distill-Qwen-7B показывает качество, близкое к o1-mini.
- **Quantization**: вместо 16-bit float используются 4-bit или 8-bit представления весов. Потеря качества минимальна (1–3%), экономия памяти — 4×.
- **GGUF формат** + Ollama: позволяет запускать quantized модели одной командой.

```bash
# Запуск DeepSeek-R1 distilled на ноутбуке
ollama run deepseek-r1:7b-q4_K_M
```

## 15.9. Что стабильно, а что изменится

### To, что меняется каждые 6 месяцев

- Конкретные модели и их возможности
- Размеры контекстных окон
- Цены на API
- Фреймворки и инструменты
- Бенчмарки и лидерборды
- Архитектурные новшества (SSM, Diffusion, hybrid)

### То, что остаётся стабильным (и описано в этой книге)

| Принцип | Почему стабилен |
|---------|-----------------|
| **Токенизация определяет бюджет** (Гл. 1) | BPE будет работать, пока есть текст |
| **Модель хранит ассоциации, не факты** (Гл. 2) | Фундаментальное свойство нейросетей |
| **Галлюцинации — режим, не баг** (Гл. 3) | Следствие функции потерь |
| **Каузальный порядок влияет на результат** (Гл. 4) | Даже Diffusion LLM нужна структура промпта |
| **Промпт — это протокол** (Гл. 6) | Структурированный вход → предсказуемый выход |
| **Разделяй генерацию и верификацию** (Гл. 10, 12) | Любая надёжная система требует проверки |
| **Делегируй вычисления инструментам** (Гл. 11) | Модели не будут «уметь» арифметику |
| **Логируй всё** (Гл. 12) | Любая система требует observability |
| **Маленький модуль лучше монолита** (Гл. 13) | Контекстное окно не исчезнет |

## 15.10. Практический вывод

### Чек-лист «Что обновить в своих системах»

| # | Действие | Приоритет |
|---|----------|-----------|
| 1 | **Внедрить model routing**: простые запросы → дешёвая модель, сложные → дорогая | Высокий |
| 2 | **Включить prompt caching** для длинных system prompts | Высокий |
| 3 | **Оценить self-hosting** для рутинных задач (vLLM + Llama 4 / DeepSeek) | Средний |
| 4 | **Протестировать reasoning-модели** на ваших сложных задачах | Средний |
| 5 | **Перейти на MCP** для интеграции инструментов | Средний |
| 6 | **Мониторить inference-стоимость** с разбивкой по модели и задаче | Высокий |
| 7 | **Следить за гибридными архитектурами** (Mamba + attention) | Низкий (пока) |
| 8 | **Тестировать SLM** для edge / low-latency сценариев | Зависит от use case |

### Ментальная модель

> **Самое качественное решение 2026 года — не «большая модель», а гибридная reasoning-модель (MoE + SSM + attention), обученная с учётом test-time compute, развёрнутая как часть агентной системы с оптимизированным inference на гибридной инфраструктуре.** Это даёт максимальное качество при разумных затратах и масштабе.

Но главное — фундамент не меняется. Протоколы, верификация, декомпозиция, инструменты, логирование — всё это будет работать и через два года, какие бы модели ни вышли.
